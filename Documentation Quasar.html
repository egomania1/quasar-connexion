Vue.js — Le fondement

Avant de comprendre Quasar, il faut comprendre Vue.js, le framework sur lequel il repose.

Qu'est-ce que Vue.js ? Vue.js (prononcé "view") est un framework JavaScript open-source pour construire des interfaces utilisateur. Créé par Evan You en 2014, il s'appuie sur HTML, CSS et JavaScript standards avec un modèle de programmation déclaratif basé sur les composants.

Historique

2014 — Vue.js 1.0, créé par Evan You (ex-Google)
2016 — Vue.js 2.0, adoption massive
2020 — Vue.js 3.0, réécriture complète
2023 — Vue 2 atteint sa fin de vie (EOL)
Aujourd'hui — Vue 3 est la version officielle

Philosophie "Progressive"

Vue est conçu pour être adopté progressivement. Il peut être utilisé comme une simple bibliothèque UI sur une page existante, ou comme un framework complet pour des SPA complexes.

Les 2 piliers de Vue.js

Rendu Déclaratif

Vue étend le HTML standard avec une syntaxe de template permettant de décrire l'interface en fonction de l'état JavaScript. Le DOM se met à jour automatiquement quand les données changent.

Réactivité

Vue surveille automatiquement les changements d'état JavaScript et met à jour efficacement le DOM. Plus besoin de manipuler le DOM manuellement.

Quasar Framework — Vue d'ensemble

Quasar est le framework Vue.js le plus complet pour créer des applications multi-plateformes.

Définition officielle Quasar est un framework open-source sous licence MIT, basé sur Vue.js, qui permet aux développeurs d'écrire le code une seule fois et de le déployer simultanément comme site web, application mobile et/ou application desktop.

Mission principale

Éliminer le besoin de maintenir plusieurs codebases pour différentes plateformes. Un seul projet Quasar peut produire une SPA, une PWA, une app mobile Android/iOS et une app desktop Windows/Mac/Linux.

Licence & Open Source

Quasar est 100% open source sous licence MIT. Gratuit pour usage personnel et commercial. Communauté active sur GitHub, Discord et le forum officiel.

Avantage clé

Seul Quasar offre un tel niveau de déploiement multi-plateforme depuis une unique base de code Vue.js, tout en maintenant des performances proches du natif.

Philosophie

Quasar regarde vers l'avenir et définit ses propres standards élevés. Il remet en question les conventions établies quand de meilleures solutions existent.

Points forts techniques

Tree-shaking auto

Seul le code réellement utilisé est inclus dans le bundle final. Zéro overhead inutile.

40+ langues

Packs de langue intégrés. Support RTL (arabe, hébreu) avec conversion CSS automatique.

Material Design

Tous les composants suivent les guidelines Material Design. Personnalisables via CSS/SASS.

Performances

Conçu avec performance et responsivité en tête. L'overhead de Quasar est à peine perceptible.

Aucune dépendance externe

Pas besoin de Hammer.js, Moment.js ou Bootstrap. Tout est inclus et optimisé.

Tests & Sécurité

Outils de test et d'audit de sécurité intégrés directement dans l'écosystème.

Plugins Quasar

Les plugins Quasar sont des services accessibles via useQuasar() ou $q. Ils doivent être déclarés dans quasar.config.js → plugins: [].

Accéder aux plugins Dans la Composition API : const $q = useQuasar()
Dans l'Options API : this.$q

Notify

Affiche des notifications toast en haut ou en bas de l'écran.

// quasar.config.js → plugins: ['Notify']

import { useQuasar } from 'quasar'
const $q = useQuasar()

// Notification simple
$q.notify('Message simple')

// Notification avancée
$q.notify({
  message: 'Fichier sauvegardé',
  color: 'positive',          // positive | negative | warning | info
  icon: 'save',
  position: 'top-right',      // top | bottom | left | right | top-left...
  timeout: 3000,
  actions: [
    { label: 'Annuler', color: 'white', handler: () => { /* ... */ } }
  ],
  multiLine: false,
  html: false
})

// Notification persistante (mise à jour)
const notif = $q.notify({ message: 'Chargement...', spinner: true, timeout: 0 })
notif({ message: 'Terminé !', spinner: false, timeout: 2000 }) // mise à jour

Dialog

Affiche des boîtes de dialogue programmatiques (alert, confirm, prompt) ou des composants Vue dans une modal.

// quasar.config.js → plugins: ['Dialog']

// Confirmation simple
$q.dialog({
  title: 'Confirmer',
  message: 'Voulez-vous supprimer cet élément ?',
  cancel: true,
  persistent: true
}).onOk(() => {
  console.log('Confirmé')
}).onCancel(() => {
  console.log('Annulé')
})

// Prompt (saisie utilisateur)
$q.dialog({
  title: 'Nom',
  prompt: {
    model: '',
    type: 'text',
    isValid: val => val.length > 2
  }
}).onOk(val => { console.log('Valeur:', val) })

// Dialog avec un composant Vue personnalisé
import MonDialog from './MonDialog.vue'
$q.dialog({ component: MonDialog, componentProps: { userId: 42 } })
  .onOk(data => console.log(data))

⏳ Loading

Affiche un overlay de chargement global sur toute la page.

Pinia — Gestion d'état

Pinia est le gestionnaire d'état officiel de Vue 3. Il remplace Vuex et est intégré par défaut dans les nouveaux projets Quasar.

Pourquoi Pinia ?

API simple et intuitive, excellent support TypeScript, DevTools intégrés, pas de mutations (contrairement à Vuex), stores composables et modulaires.

Installation dans Quasar

Pinia est automatiquement configuré lors de la création d'un projet. Les stores se placent dans /src/stores/.

Créer un store

// src/stores/user.js
import { defineStore } from 'pinia'
import { ref, computed } from 'vue'

// Style "Setup Store" (recommandé avec Vue 3)
export const useUserStore = defineStore('user', () => {

  // State
  const user = ref(null)
  const isLoggedIn = ref(false)

  // Getters (computed)
  const displayName = computed(() =>
    user.value ? user.value.name : 'Invité'
  )

  // Actions
  async function login(email, password) {
    const data = await apiLogin(email, password)
    user.value = data.user
    isLoggedIn.value = true
  }

  function logout() {
    user.value = null
    isLoggedIn.value = false
  }

  return { user, isLoggedIn, displayName, login, logout }
})

Utiliser le store dans un composant

<template>
  <q-page class="q-pa-md">
    <div v-if="userStore.isLoggedIn">
      Bonjour {{ userStore.displayName }}
      <q-btn label="Déconnexion" @click="userStore.logout()" />
    </div>
    <div v-else>
      <q-btn label="Connexion" @click="userStore.login(email, pass)" />
    </div>
  </q-page>
</template>

<script setup>
import { useUserStore } from 'src/stores/user'
const userStore = useUserStore()
</script>

Déstructurer sans perdre la réactivité — storeToRefs

import { storeToRefs } from 'pinia'
import { useUserStore } from 'src/stores/user'

const userStore = useUserStore()

//  Mauvais — perd la réactivité
const { user, isLoggedIn } = userStore

//  Correct — réactif grâce à storeToRefs
const { user, isLoggedIn, displayName } = storeToRefs(userStore)
// Les actions n'ont pas besoin de storeToRefs
const { login, logout } = userStore

Routing — Vue Router

Quasar intègre Vue Router. La configuration des routes se trouve dans src/router/routes.js.

Configuration des routes

// src/router/routes.js
const routes = [
  {
    path: '/',
    component: () => import('layouts/MainLayout.vue'),
    children: [
      { path: '', component: () => import('pages/IndexPage.vue') },
      { path: 'about', component: () => import('pages/AboutPage.vue') },

      // Route dynamique
      { path: 'users/:id', component: () => import('pages/UserPage.vue') },

      // Route avec meta (utilisée pour les guards)
      {
        path: 'admin',
        component: () => import('pages/AdminPage.vue'),
        meta: { requiresAuth: true, roles: ['admin'] }
      }
    ]
  },

  // Route 404
  { path: '/:catchAll(.*)*', component: () => import('pages/ErrorNotFound.vue') }
]

export default routes

Navigation Guards (protection des routes)

// src/router/index.js
import { route } from 'quasar/wrappers'
import { createRouter, createWebHistory } from 'vue-router'
import routes from './routes'

export default route(function({ store }) {
  const Router = createRouter({
    history: createWebHistory(),
    routes
  })

  // Guard global — exécuté avant chaque navigation
  Router.beforeEach((to, from) => {
    const authStore = useAuthStore(store)

    if (to.meta.requiresAuth && !authStore.isLoggedIn) {
      return { path: '/login', query: { redirect: to.fullPath } }
    }
  })

  return Router
})

Boot Files

Les boot files s'exécutent avant le montage de l'application Vue. Ils initialisent les plugins, configurent des services globaux (axios, i18n, etc.) et sont déclarés dans quasar.config.js → boot: [].

Structure d'un boot file Chaque boot file est dans src/boot/ et exporte une fonction defineBoot qui reçoit { app, router, store }.

Exemple — axios.js

// src/boot/axios.js
import { defineBoot } from '#q-app/wrappers'
import axios from 'axios'

const api = axios.create({
  baseURL: 'https://mon-api.com/api/v1',
  timeout: 10000,
  headers: { 'Content-Type': 'application/json' }
})

// Intercepteur — ajouter le token JWT automatiquement
api.interceptors.request.use(config => {
  const token = localStorage.getItem('token')
  if (token) config.headers.Authorization = `Bearer ${token}`
  return config
})

// Intercepteur — gérer les erreurs globalement
api.interceptors.response.use(
  response => response,
  error => {
    if (error.response?.status === 401) {
      // Token expiré → rediriger vers login
    }
    return Promise.reject(error)
  }
)

export default defineBoot(({ app }) => {
  // Rend $api accessible dans tous les composants
  app.config.globalProperties.$api = api
})

export { api }  // export pour utilisation dans les stores/composables

Exemple — i18n.js (internationalisation)

// src/boot/i18n.js
import { defineBoot } from '#q-app/wrappers'
import { createI18n } from 'vue-i18n'

const messages = {
  fr: {
    welcome: 'Bienvenue',
    buttons: { save: 'Sauvegarder', cancel: 'Annuler' }
  },
  en: {
    welcome: 'Welcome',

  }
}

const i18n = createI18n({
  locale: 'fr',
  fallbackLocale: 'en',
  messages,
  legacy: false  // nécessaire pour Composition API
})

export default defineBoot(({ app }) => {
  app.use(i18n)
})

// Utilisation dans un composant
// import { useI18n } from 'vue-i18n'
// const { t } = useI18n()
// t('welcome')  →  "Bienvenue"

    { path: 'firebase', server: false },  // côté client seulement (SSR)
    { path: 'analytics', server: false }
  ]
}))
Ordre d'exécution Les boot files s'exécutent dans l'ordre déclaré dans le tableau boot. Mettez d'abord les dépendances (ex: auth avant analytics).
SSR — client vs serveur Certains plugins (analytics, localStorage) ne fonctionnent que côté client. Utilisez { path: 'plugin', server: false } pour les exclure du rendu serveur.

Configuration — quasar.config

Le fichier quasar.config.js est le cœur de la configuration de votre projet.

// quasar.config.js
import { defineConfig } from '#q-app/wrappers'

export default defineConfig(ctx => {
  return {
    css: ['app.scss'],
    boot: ['axios', 'i18n'],
    extras: ['material-icons', 'roboto-font'],

    framework: {
      lang: 'fr',
      iconSet: 'material-icons',
      components: ['QBtn', 'QCard', 'QTable'],
      plugins: ['Notify', 'Loading', 'Dialog', 'Dark'],
      directives: ['Ripple', 'ClosePopup', 'Intersection'],
      config: {
        dark: 'auto',
        notify: { position: 'top-right', timeout: 3000 }
      }
    },

    animations: 'all',

    devServer: {
      port: 9000,
      open: true,
      proxy: {
        '/api': { target: 'http://localhost:3000', changeOrigin: true }
      }
    },

    build: {
      vueRouterMode: 'history',
      distDir: 'dist/spa',
      minify: true,
      sourcemap: ctx.dev,
      env: {
        API_URL: ctx.dev ? 'http://localhost:3000' : 'https://api.monsite.com'
      }
    }
  }
})

Options de configuration détaillées

css

Charge des fichiers CSS/SCSS globaux depuis /src/css/. Appliqués à toute l'application.

boot

Liste des fichiers de démarrage dans /src/boot/. Exécutés avant le montage du composant racine Vue.

extras

Importe des polices et jeux d'icônes. Ex: 'material-icons', 'fontawesome-v6', 'mdi-v7'.

framework

Configure la langue, les icônes, importe sélectivement composants/plugins/directives Quasar (optimise le bundle).

devServer

Configure le serveur Vite local : port, HTTPS, proxy pour les appels API, ouverture auto du navigateur.

build

Gère la production : mode router (hash/history), dossier de sortie, minification, sourcemaps, variables d'environnement.

Structure du projet

Arborescence standard d'un projet Quasar.

mon-projet/

 src/                     # Code source principal
    assets/             # Images, fonts, fichiers statiques
    boot/               # Fichiers de démarrage (axios, i18n...)
    components/          # Composants Vue réutilisables
    composables/         # Composition functions (logique réutilisable)
    css/                 # Styles globaux (app.scss, variables.scss)
    layouts/             # Layouts de l'app (MainLayout.vue...)
    pages/               # Pages/vues de l'application
    router/              # Configuration Vue Router
    stores/              # Stores Pinia (gestion d'état)
    App.vue              # Composant racine

 public/                  # Fichiers publics (copiés tel quel)
 src-electron/            # Code spécifique Electron (si mode ajouté)
 src-capacitor/           # Code spécifique Capacitor (si mode ajouté)
 src-pwa/                 # Service Worker PWA (si mode ajouté)
 src-ssr/                 # Code serveur SSR (si mode ajouté)
 src-bex/                 # Extension navigateur (si mode ajouté)

 quasar.config.js         # Configuration centrale Quasar
 package.json
 vite.config.js           # Config Vite (si nécessaire)
Layouts vs Pages Les Layouts définissent la structure globale (header, footer, drawer). Les Pages s'insèrent dans les layouts et représentent les différentes routes de l'application.
Boot files Les fichiers boot (/src/boot/) initialisent des plugins ou configurations avant le montage de Vue. Ex: configurer axios avec un base URL, initialiser i18n, etc.

Quand choisir Quasar ?

Projet multi-plateforme

Vous voulez cibler web + mobile + desktop avec une seule base de code. C'est le cas d'usage idéal de Quasar.

Productivité maximale

Vous voulez démarrer vite avec une bibliothèque de composants complète sans configurer Tailwind, Vuetify, etc. séparément.

Équipe Vue.js

Votre équipe connaît déjà Vue.js. Quasar s'intègre parfaitement sans changer les habitudes.

SEO pur / Blog

Pour un projet 100% SSR/SEO, Nuxt.js peut être plus adapté avec son écosystème spécialisé.

App mobile ultra-native

Pour des performances mobiles maximales avec accès total aux APIs natives, React Native ou Flutter peuvent être plus appropriés.
